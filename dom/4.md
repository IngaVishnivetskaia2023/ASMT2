## How the DOM is created in the browser under the hood, and how standards and data flow relate:

#### 1. Bytes to Characters
- The browser receives raw bytes from the network (e.g., over HTTP).
- It uses the specified character encoding (like UTF-8) to decode these bytes into characters (text).

#### 2. Characters to Tokens
- The browser’s HTML parser reads the character stream and breaks it into tokens (e.g., start tags <div>, end tags </div>, text nodes, comments).
- This process follows the W3C HTML5 specification, which defines exactly how to handle every possible sequence of characters (including errors and edge cases).

#### 3. Tokens to Nodes
- The parser uses these tokens to build nodes (elements, text nodes, comments, etc.).
- Nodes are created according to the rules in the HTML5 spec (e.g., how to handle nested tags, missing tags, etc.).

#### 4. Nodes to DOM Tree
- The nodes are organized into a tree structure—the Document Object Model (DOM).
- The DOM is a live, in-memory tree of objects representing the structure and content of the HTML document.
- Each node in the tree corresponds to an element, text, or comment in the original HTML.

#### 5. W3C and HTML5 Standards
The W3C (World Wide Web Consortium) and WHATWG define the HTML and DOM specifications.
Browsers implement these specs to ensure consistent parsing and DOM creation across platforms.

#### Layout
The layout stage is where the browser figures out exactly where and how big every visible element is, using the render tree and CSS rules, before painting pixels to the screen.

| Step | Description |
| :-- | :-- |
| Build Render Tree | Combine DOM + CSSOM, exclude non-visual elements |
|  |  |
| Determine Contexts | Block, inline, flex, grid, etc. |
| Calculate Boxes | Compute size, position, margins, borders, padding |
| Resolve Dependencies | Parent/child size dependencies, recursive calculation |
| Special Layouts | Flexbox, grid, floats, tables |
| Viewport/Scrolling | Handle overflow, scrollbars |
| Output Coordinates | Final positions/sizes for painting |

#### Technical Steps of the Layout Stage

🌴 **Input: Render Tree:**
The browser first builds the render tree by combining the DOM and CSSOM.
The render tree contains only visible elements (excluding those with display: none, etc.).

🌴 **Formatting Contexts:**
The browser determines the formatting context for each element (block, inline, flex, grid, table, etc.).
This context affects how child elements are laid out.

🌴 **Flow Calculation:**
The browser traverses the render tree top-down.
For each element, it calculates:
- Box model properties: width, height, margin, border, padding
- Position: top, left, right, bottom (for positioned elements)
- Stacking context: z-index, stacking order

🌴 **Dependency Resolution:**
Some elements depend on their children (e.g., a container with height: auto).
The browser may need to recursively calculate children before parents.

🌴 **Handling Special Layouts:**
- Floats: Elements floated with float are taken out of normal flow and affect siblings.
- Flexbox/Grid: The browser applies flex/grid algorithms to distribute space and align items.
- Tables: Table layout is handled with special rules for rows, columns, and cells.

🌴 **Viewport and Scrolling:**
The browser determines the size of the viewport and calculates scrollbars if content overflows.

🌴 **Fragmentation:**
For paged media (print, columns), the browser may split content into fragments.

🌳 **Output: Coordinates**
The result is a set of **exact coordinates and sizes for every box** in the render tree.

This information **is used in the next stage: painting**.

### Render Blocking Resources

| Resource Type | Render-Blocking by Default? | Notes |
| :-- | :-- | :-- |
| CSS | Yes | Blocks rendering until loaded/parsed |
| JS (<script>) | Yes (unless defer/async) | Blocks parsing and rendering |
| Images | No | Page can render before images load |


#### Why is CSS render-blocking?
- The browser needs the CSSOM (CSS Object Model) to know how elements should look and be laid out.
- If the browser painted the page before CSS was loaded, users would see unstyled or incorrectly laid out content ("flash of unstyled content" or FOUC).
Therefore, the browser waits for all CSS to be loaded and parsed before proceeding to layout and paint.


| Where media is set | Render-blocking? | When CSS is loaded/applied |
| :-- | :-- | :-- |
| @media **inside CSS file** | Yes | Always loaded/parsed, rules applied as needed |
| media attribute on `<link>` | No (if media doesn't match) | Only loaded/applied when media matches |




#### Exceptions / Optimizations
- CSS loaded with `media="print"` or other non-matching media queries **is not render-blocking** for the initial screen.
- CSS loaded with `media="none"` and then switched to `media="all"` can be used for non-blocking loading (advanced optimization).
- Critical CSS inlined in `<style>` in the `<head>` can help render above-the-fold content faster.

#### Conclusion

If you add media attributes to the `<link>` tag, you can turn your CSS files from being “render blocking” into a “non-render blocking” one, which will shorten the critical rendering path. You can use this approach if your site is small, or if shortening the CRP time is your top priority (i.e. you don't worry too much about multiple HTTP requests and/or maintainability).

The downside is that every time you want to change a global CSS rule, you have to make the change on multiple files (because remember, each media rule references an individual CSS file).

Which approach you decide on is really up to you, and depends largely on how large your site is.

### Question #1: Why did the first paint happen before the DOMContentLoaded event?
The **first paint** can happen before the `DOMContentLoaded` event because the **browser paints content incrementally** as it parses the HTML and CSS. Browsers are optimized to render visible content as soon as possible, even if the **DOM** and **CSSOM** trees are incomplete.

#### How does this work?
The browser doesn’t wait for the entire DOM or CSSOM to be built before starting to paint. 
Instead, it paints in small chunks as soon as enough information is available to display part of the page.
Each browser has its own internal (and often proprietary) strategies for optimizing the painting process.

#### Why does this matter?
This is why you might see multiple paint events during the rendering process.

### Question #2: What does the DOMContentLoaded event mean?
The DOMContentLoaded event is a browser event that fires when:
- The HTML document is fully parsed and the DOM tree is built.
- All inline JavaScript and **blocking** `<script>` tags in the HTML have been executed.

#### How to use it in your code?
You can listen for the `DOMContentLoaded` event on the `document` object.

### How to Approach JavaScript render blocking?

#### Option 1: placing JavaScript at the bottom of your page
Placing JavaScript at the bottom of the page is a traditional technique that was commonly used in vanilla JavaScript development to improve page load performance. This approach was especially relevant before modern script-loading attributes like `defer` and `async` became widely supported. 
In vanilla JavaScript (or older web development practices), placing `<script>` tags at the bottom of the HTML document (just before the closing `</body>` tag) was a way to prevent JavaScript from blocking the Critical Rendering Path (CRP) as by default, `<script>` tags are synchronous and blocking.

#### Option 2: Using `async`
Ensures that the script is downloaded in parallel with HTML parsing and executed as soon as it is ready, without waiting for HTML parsing to complete.
- Once the script is downloaded, it is executed **immediately**, even if the HTML parsing is not yet complete.
- This can happen **before, during, or after** the `DOMContentLoaded` event, depending on when the script finishes downloading.
- So such script execution may happen any time before `load` event ❗

#### Option 3: Using `defer`
If defer is set for classic script, the script is downloaded in parallel but executed after the HTML parsing is complete and before the `DOMContentLoaded` event fires. For module scripts, `defer` is unnecessary because they are already non-blocking and deferred.

#### Option 4: Custom event listener
Setting custom event listeners for `load` or `DOMContentLoaded` events is still technically valid and can be useful in certain scenarios, 
but it is **less common in modern web development** due to the availability of better practices and modern tools.
- Frameworks like React, Vue, and Angular handle DOM rendering and updates internally.
- These frameworks ensure that the DOM is ready before rendering components, so developers rarely need to manually listen for `DOMContentLoaded`.
- Tools like Webpack, Vite, and Parcel generate optimized JavaScript bundles and often include configurations that ensure scripts are executed at the right time.
Developers typically rely on these tools to handle script loading and execution.

#### Option 5: Using preload
Using `<link rel="preload" as="script">` is still a valid and useful approach in modern web development, but its use case is specific and should be applied strategically. It is particularly helpful for optimizing the Critical Rendering Path (CRP) and improving page load performance when used correctly.

- The browser starts downloading the script as soon as it encounters the `<link rel="preload" as="script">` tag.
- When the browser later encounters the `<script>` tag for the same resource, it can use the **already-downloaded script**, avoiding the need to fetch it again.
- It reduces the time to download critical scripts, improving the **First Contentful Paint (FCP)** and **Time to Interactive (TTI)**.
- It is particularly useful for large JavaScript bundles or scripts that **are critical for rendering above-the-fold content**.

❗Even with speculative parsing, **scripts are still synchronous by default** unless they are explicitly marked with `async`, `defer`, or `type="module"`. This means that classic scripts (without these attributes) are executed in the order they appear in the HTML document, and they block further HTML parsing until they are fully executed.

#### How It Works:
- While the main thread is busy executing JavaScript or performing other tasks, the browser's **speculative parser scans the rest of the HTML document** to identify resources that need to be fetched (e.g., `<script>`, `<link>`, `<img>`).
- These resources are downloaded in parallel, but their execution or application (e.g., applying styles or running scripts) still follows the normal rules.
