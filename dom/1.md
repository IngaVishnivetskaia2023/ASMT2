## DOM

https://developer.chrome.com/blog/inside-browser-part3

DOM is an object representation of HTML (or XML), and **it is created by the browser engine by parsing the HTML document it receives**. 
The DOM is language-agnostic, but JavaScript is the most commonly used language to interact with it in the browser.

#### В DOM наследуют EventTarget:

- Element (все HTML- и SVG-элементы)
- Document
- Window
- Некоторые другие специальные объекты (например, `IDBRequest`, `AudioNode`)

#### В DOM НЕ наследуют EventTarget:

- Text (текстовые узлы)
- Comment (комментарии)
- Некоторые внутренние или вспомогательные объекты

```
const div = document.createElement('div');
console.log(div instanceof EventTarget); // true

const text = document.createTextNode('hello');
console.log(text instanceof EventTarget); // false

const comment = document.createComment('note');
console.log(comment instanceof EventTarget); // false

```

В DOM только те объекты, которые **могут быть целью событий** (например, элементы, документ, окно), наследуют методы EventTarget.
Текстовые и комментарные узлы — нет.

### Event Delegation

Event delegation is a technique where **a single event listener is attached to a parent element**, 
and it handles events for its child elements by relying on event bubbling. 
When an event occurs on a child, it bubbles up to the parent, allowing the parent’s listener to detect and process the event for any child.

#### Technical summary:

- Attach one listener to a common ancestor.
- Use event’s `.target` property to identify **which child triggered the event**.
- Works because events **bubble up** from child to parent in the DOM.

### Critical Rendering Path in SPA

#### Parsing HTML → Building the DOM
- The browser receives the HTML document from the server and begins parsing it.
- The HTML is converted into characters, then tokens, and finally nodes to build the DOM tree incrementally.
- The DOM tree represents the structure of the document (e.g., `<div id="root">`).
- When the browser encounters a **blocking** `<script>` tag (e.g., `<script src="bundle.js"></script>`), it **pauses HTML parsing until the script is downloaded** and executed.

This is critical for SPAs **because the JavaScript bundle contains the logic to dynamically generate the DOM** (e.g., React rendering the UI into `<div id="root">`).

#### Why Script Placement Matters:
If the `<script>` tag is placed in the `<head>` without `async` or `defer`, it **blocks parsing early in the process, delaying the creation of the DOM**.
Best practice for SPAs: Place the `<script>` tag at the end of the `<body>` to ensure the browser parses as much HTML as possible before pausing.


#### Parsing CSS → Building the CSSOM
- The browser downloads and parses CSS files **in parallel** with HTML parsing.
- The parsed CSS is converted into the CSSOM tree, which represents the styles applied to the DOM elements.
- Media queries are evaluated, and invalid CSS is quietly skipped.

#### Impact of CSS Blocking:
- CSS is render-blocking, meaning the browser cannot paint anything on the screen until the CSSOM is ready.
- This ensures that the page is styled correctly before it is displayed.

#### Executing JavaScript
- The browser downloads and executes JavaScript, which can modify both the DOM and CSSOM.
- JavaScript execution **blocks HTML parsing** unless the `<script>` tag includes `async` or `defer`.

#### Building the Render Tree
- The browser combines the DOM tree and CSSOM tree to create the Render Tree.
- The Render Tree includes only visible elements:
- Elements with `display: none` are **excluded**.
- Elements with `visibility: hidden` are **included (but not painted)**.
- Scripts, meta tags, and `<head>` elements are ignored in the Render Tree.

❗In SPAs, the Render Tree is created **after** React commits the Virtual DOM to the real DOM.


#### Layout (Reflow)
- The browser calculates the size and position of every element in the Render Tree.
- This step **is CPU-intensive** and traverses the Render Tree top-down.
- Layout is triggered whenever layout-affecting properties (e.g., `width`, `height`, `position`) change.

🐉 Use `transform` instead of `top/left` for animations to avoid layout recalculations

#### Paint
- The browser fills pixels on the screen, including colors, text, borders, and shadows.
- Painting is **CPU-intensive and may create multiple layers for stacking contexts** (e.g., z-index).

🦖 Use `opacity` instead of `visibility` for fades to avoid triggering layout changes

#### Composition
- The browser merges layers in the correct order using the GPU.
- Compositing is the **fastest stage** and enables smooth animations when optimized properly.

🍹 Use `will-change` on frequently animated elements to optimize compositing





### Real DOM vs Virtual DOM

| Type | Where? | Visible? | Who manages? | Used for? |
| :-- | :-- | :-- | :-- | :-- |
| Real DOM | Browser memory | Yes | Browser | Rendering, interaction |
| Virtual DOM | JS memory | No | React | Efficient updates, diff |



| Aspect | Browser Memory | JS Memory |
| :-- | :-- | :-- |
| Managed by | Browser (includes JS engine) | JS engine (part of browser) |
| Contains | DOM, CSSOM, rendering, JS engine, etc. | JS variables, objects, virtual DOM |
| Visible UI connection | Yes (DOM, rendering) | No (unless JS updates DOM) |
| Scope | Whole browser process | JS execution context |
