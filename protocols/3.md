| **Метод**             | **Плюсы**                                                                 | **Минусы**                                                | **Типичные сценарии**                |
|------------------------|--------------------------------------------------------------------------|----------------------------------------------------------|---------------------------------------|
| **Basic Authorization** | Простая реализация.                                                     | Небезопасна. Легко перехватить.                          | Тестовые API или внутренние задачи.  |
| **Bearer Token**        | Гибкий. Совместим с SPA и мобильными приложениями.                     | Уязвим к XSS при хранении в `localStorage`.              | REST API, микросервисы.              |
| **HTTP-Only Cookies**   | Защита от XSS.                                                         | Подвержен CSRF, если не настраивать защиту.              | Веб-приложения, защищённые платформы.|
| **OAuth2**              | Интеграция с третьими сервисами. Простота для пользователей.           | Сложная настройка. Зависимость от провайдеров.           | Сайты соцсетей, подписки.            |
| **Session-Based**       | Привязан к устройству/сессии. Легко управляем.                         | Зависимость от состояния сервера (сложно масштабировать). | Традиционные платформы (e-commerce). |
| **API Key**             | Просто настроить и реализовать.                                        | Нет встроенной безопасности, ключи легко украсть.        | Публичные API.                       |


- Для API: Использовать Bearer Token (JWT) или OAuth2, если требуется третья сторона.
- Для браузерных приложений: Использовать JWT с HTTP-Only Cookies для большего уровня защиты.
- Для быстрых интеграций: API-ключи, если данные не критичны.
- Для традиционных сайтов: Сессии, если масштабируемость не является проблемой.
